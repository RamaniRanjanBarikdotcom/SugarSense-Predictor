01 Abstract

Abstract

SugarSense Predictor is a full-stack intelligent web application designed for early-stage diabetes risk screening. The system combines a Support Vector Machine (SVM) classifier trained on the Pima Indians Diabetes Dataset with a multi-provider AI health chatbot to deliver accurate, accessible, and actionable diabetes risk assessments. The frontend is built as a React Single Page Application (SPA) served by a Flask Python backend, providing an intuitive interface for entering clinical health metrics and receiving real-time predictions. The integrated AI chatbot — powered by Claude, Gemini, GPT, or open-source models via OpenRouter — delivers evidence-based health guidance through a structured knowledge base of over 110 Q&A pairs. The system achieves approximately 77–78% prediction accuracy on test data and demonstrates how machine learning and generative AI can together improve chronic disease awareness and patient engagement.

02 Introduction

Diabetes mellitus is one of the fastest-growing chronic diseases globally, affecting over 537 million adults worldwide as of 2021, with projections exceeding 700 million by 2045 (IDF Diabetes Atlas). Despite its prevalence, diabetes often goes undetected until it reaches advanced stages, by which time complications such as neuropathy, nephropathy, and cardiovascular disease may already be present. Early detection is therefore critical to improving long-term patient outcomes and reducing healthcare costs.

Traditional diabetes screening relies on periodic laboratory tests, which can be costly and inaccessible in low-resource settings. Machine learning (ML) offers a promising alternative: by analyzing combinations of non-invasive clinical measurements, ML models can identify patterns that indicate diabetes risk before formal diagnosis is possible.

SugarSense Predictor addresses this gap by providing a web-based diabetes risk screening tool that is fast, accessible, and intelligent. The application integrates an SVM classifier with a multi-turn AI health chatbot, allowing users to not only receive a risk prediction but also ask follow-up questions and receive tailored health guidance — all within a single, unified interface.

The system is designed for two primary audiences: (1) individuals who wish to screen themselves for diabetes risk using publicly available clinical data points, and (2) healthcare students and professionals who want a demonstration of how ML and generative AI can be combined in a clinical screening context.

Problem Statement

Existing diabetes risk tools are either too simplistic (basic questionnaires) or too complex (laboratory-dependent). There is a need for an intelligent, easy-to-use system that can assess diabetes risk from commonly available health metrics and provide personalized health guidance without requiring medical expertise from the user. Additionally, most existing tools do not explain their predictions or provide actionable follow-up information.

07 Machine Learning Pipeline

Training Pipeline

1
Data Loading Load diabetes1.csv (768 rows × 9 columns) using pandas. Separate features X (8 columns) and target y (Outcome column).
2
Train-Test Split Split dataset into 80% training (614 samples) and 20% testing (154 samples) using stratified sampling to preserve class balance.
3
Feature Scaling Fit StandardScaler on training data, transform both training and test sets. Scaler serialized to sc1.pkl for inference.
4
SVM Training Train SVC with linear kernel on scaled training data. Model learns a hyperplane maximizing the margin between diabetic and non-diabetic feature vectors.
5
Evaluation Evaluate on held-out test set: accuracy ~77–78%, with additional metrics (precision, recall, F1, confusion matrix) logged in the Jupyter notebook.
6
Serialization Save trained SVM to classifier1.pkl and scaler to sc1.pkl using Python's pickle module. Both artifacts are loaded once at Flask server startup.

Inference Pipeline

# 1. Receive 8 values from /predict_api JSON payload features = [pregnancies, glucose, bloodPressure, skinThickness, insulin, bmi, diabetesPedigreeFunction, age] # 2. Shape into numpy array model_input = np.array([features], dtype=float) # shape (1, 8) # 3. Apply pre-fitted StandardScaler scaled_input = sc.transform(model_input) # mean=0, std=1 # 4. SVM predicts class label prediction = int(model.predict(scaled_input)[0]) # 0 or 1 # 5. Return JSON response return jsonify({"prediction": prediction})

Input Validation

All 8 fields are validated server-side for: presence (required), data type (must parse as float), and value bounds (per-field min/max). Invalid requests return HTTP 400 with a descriptive error message. The JSON API and form API use separate validation paths, both enforcing the same rules.

AI Provider Routing Logic

m = _CHAT_MODEL # from .env if m == "claude" or m.startswith("claude-"): → _call_claude() # Anthropic SDK elif m.startswith("gpt-", "o1", "o3", "o4"): → _call_openai() # OpenAI REST elif m.startswith("gemini-"): → _call_google() # Google REST elif m.startswith("hf/"): → _call_huggingface() # HF REST else: → _call_openrouter() # OpenRouter REST

11 AI Chatbot System

Design Overview

The Health Assistant chatbot is a full multi-turn conversational AI embedded as a floating panel in the React SPA. It is powered by state-of-the-art large language models with a structured system prompt that includes domain-specific diabetes knowledge.

System Prompt Structure

The AI system prompt has two layers:

A
Role & Behaviour Instructions Defines the assistant as a knowledgeable, empathetic Health Assistant. Sets rules: evidence-based responses, non-alarmist tone, always remind users results are not medical diagnoses, direct emergencies to 911, never prescribe medications.
B
Health Knowledge Base (110+ Q&A pairs) Loaded from health_knowledge.json at Flask startup. Formatted as structured markdown and appended to the system prompt, giving the AI immediate access to detailed diabetes facts without requiring external lookups.

Prompt Caching (Claude)

When using Claude as the AI provider, the system prompt (including the full knowledge base) is sent with cache_control: {"type": "ephemeral"}. This instructs Anthropic's API to cache the system prompt prefix for up to 5 minutes, significantly reducing latency and cost for repeated chat sessions where the system prompt is unchanged.

Multi-Turn Conversation

The entire conversation history is sent with every request as a history array. Flask strips any leading assistant messages before passing to the LLM (required because Claude mandates user-first message ordering). This enables contextually aware responses across multiple turns.

Quick Prompts

Five suggested questions appear at the top of the chat for the first three messages:

  • What is diabetes?
  • How to lower blood sugar?
  • What does BMI mean?
  • Best foods to avoid?
  • What are the symptoms?

14 Conclusion

SugarSense Predictor demonstrates how modern machine learning and generative AI can be combined into a cohesive, user-friendly healthcare screening tool. By deploying a trained SVM classifier alongside a multi-provider AI chatbot within a full-stack web application, the project fulfills its core objectives: delivering fast, evidence-based diabetes risk assessment with immediate health guidance.

The project successfully integrates a proven ML model (Support Vector Machine with linear kernel) trained on the widely referenced Pima Indians Diabetes Dataset, achieving ~78% accuracy consistent with published literature. The multi-provider AI chatbot architecture — supporting five different AI providers (Anthropic, OpenAI, Google, HuggingFace, OpenRouter) through a single environment variable — demonstrates a flexible, production-grade design pattern applicable to any LLM-integrated application.

The structured health knowledge base (110+ Q&A pairs embedded in the AI system prompt) ensures the chatbot provides factually grounded, diabetes-specific guidance rather than generic responses. The combination of prompt caching, multi-turn conversation history, and graceful error handling across all five providers results in a robust and responsive user experience.

From a software engineering perspective, the project demonstrates clean separation of concerns across all three tiers: a React SPA with component-based state management, a Flask REST API with provider-agnostic routing, and a modular AI layer easily extended with new providers. The environment-variable configuration pattern ensures the system can be deployed in any context by changing a single file.

While certain features — trend graphs, real probability scores, user authentication, and alert systems — remain as future work, the current implementation provides a complete, functional, and extensible foundation for an intelligent diabetes screening assistant.
